\section{模块设计}

\subsection{模块划分原则}
本系统的模块划分不按代码目录机械展开，而是按业务职责、数据边界和协作关系进行组织。其核心目标是将当前代码中已经存在的用户、组织、资源、检测、审核、通知和日志能力沉淀为稳定模块，同时为论文检测、评审文本检测和统一报告等迭代需求预留扩展空间。

在划分过程中遵循以下原则：
\begin{itemize}[leftmargin=2em]
    \item \textbf{围绕业务能力划分模块}。模块对应稳定业务职责，而不是简单对应前端页面或后端文件夹；
    \item \textbf{高内聚、低耦合}。单个模块内部尽量完成同一类业务闭环，不直接依赖过多外部模块细节；
    \item \textbf{统一入口、差异化实现}。资源管理、检测任务管理和结果输出尽量统一，不同检测对象的差异通过专用检测模块或适配器处理；
    \item \textbf{兼顾现有实现与后续演进}。模块设计既要能映射当前 Django 后端、Vue 前端、Celery 任务和 AI 服务实现，也要能支撑下一阶段新增论文与评审能力。
\end{itemize}

\subsection{用户与权限模块（SD-YHQX-0001）}

\subsubsection{模块目标}
落实迭代需求中管理端“用户管理”目标：区分管理员与普通用户，按角色配置可执行操作，保证系统可管可控与安全性；为用户端上传、检测、报告与人工审核等能力提供统一身份与权限判定入口。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 维护平台用户账户（注册、资料、头像、邮箱验证与密码找回等），并与 Django 认证体系对接；
    \item 管理角色模型（如管理员、发布者、审核者等）及与业务动作绑定的细粒度权限（如上传、提交检测、发布、参与审核等），在接口层统一鉴权；
    \item 签发与校验访问令牌与会话，向其他模块暴露当前用户身份、角色与权限摘要，支撑操作审计与资源访问边界。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 用户注册、登录、登出与令牌刷新，登录时可选择业务角色以匹配前端与权限策略；
    \item 用户资料维护、头像上传、个人检测配额展示（与是否归属组织及组织策略相关）；
    \item 管理端用户列表与检索、用户创建与编辑、按权限位筛选、用户删除与管理员专用账户维护；
    \item 管理端为用户增减或调整业务权限、查看用户操作日志（与统计日志模块协同）；
    \item 密码重置流程（邮箱验证码发放、校验与密码更新）。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 注册与登录凭证（用户名、邮箱、密码、角色选择等）及令牌刷新请求；
    \item 用户资料变更、头像文件、密码重置邮箱与验证码；
    \item 管理端操作用的管理员身份及目标用户标识、权限字段或筛选条件。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 认证成功后的访问令牌、刷新令牌及基础用户信息（含角色、权限摘要、组织标识等）；
    \item 用户详情、分页用户列表、操作日志查询结果及权限变更结果提示；
    \item 鉴权失败或无权访问时的统一错误响应，供网关与各业务视图复用。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{User}：扩展自 \texttt{AbstractUser}，承载用户名、邮箱、角色、组织外键、头像与简介等，并通过整型 \texttt{permission} 字段编码上传、提交、发布、审核等能力位；
    \item 令牌与验证码相关状态：访问令牌与刷新令牌由认证中间层管理；密码重置验证码及过期时间可存于用户记录；
    \item 与发布者--审核者协作相关的关联关系对象（如多对多经中间表），支撑业务侧“谁可为谁审核”的约束，仍属用户域数据。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 用户提交注册或登录请求，模块校验凭据与角色合法性，通过后签发令牌并返回用户上下文；
    \item 业务请求到达时，从令牌解析用户身份，结合角色与 \texttt{permission} 判定是否允许访问目标接口或资源；
    \item 管理端发起用户或权限变更时，校验操作者具备管理员或超级管理员资格，再更新 \texttt{User} 及关联字段；
    \item 密码重置时生成限时验证码并发送邮件，校验通过后更新密码并清除验证码状态；
    \item 登出或刷新流程使旧令牌失效或轮换刷新令牌，降低凭证泄露风险。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 对前端用户端、管理端暴露 REST 风格用户与认证接口（注册、登录、登出、资料、令牌刷新、密码重置等）；
    \item 与组织与成员模块协作：读取用户所属 \texttt{Organization}，用于登录响应展示与后续组织级配额、策略校验；
    \item 为资源管理、检测任务、人工审核、报告下载等模块提供统一鉴权钩子或中间件，避免各模块重复解析令牌；
    \item 与统计日志模块协作：关键管理操作写入操作日志，便于审计与问题排查。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 密码与敏感字段必须加密存储，令牌具备有效期并优先采用短期访问令牌加长周期刷新令牌组合；
    \item 角色默认值与权限位变更规则应集中维护，避免业务代码散落硬编码导致管理端与普通用户行为不一致；
    \item 管理员能力与普通用户能力严格分离，超级管理员操作应单独约束并留痕；
    \item 权限判定除用户自身字段外，需与组织归属一致考虑（无组织用户与组织成员在配额等策略上可能不同），具体组织规则由组织与成员模块实现，本模块消费其上下文；
    \item 接口错误信息应区分认证失败与授权失败，便于前端提示与安全审计。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 引入更细 RBAC（角色--权限--资源）或策略引擎，支持按组织、按资源类型动态授权；
    \item 支持单点登录、多因素认证与企业目录对接，满足高校或期刊规模化部署需求。
\end{itemize}

\subsection{组织与成员模块（SD-ZZCY-0001）}

\subsubsection{模块目标}
支撑平台以组织为单位的账户治理与资源隔离：完成组织入驻与审批、成员加入与角色约束，并在组织维度管理检测调用配额，与迭代目标中高并发、多用户协作及可运营性要求一致。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 受理组织入驻申请，支持管理员审批、拒绝与归档，并在通过后创建正式组织及初始管理员账户；
    \item 维护组织主数据（名称、联系邮箱、描述、标识物料等）及组织与成员之间的归属关系；
    \item 维护组织级 LLM 与非 LLM 检测剩余次数等配额信息，为检测任务调度提供扣减与查询依据，并支持管理员调整组织维度权限模板。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 组织申请提交（组织信息、管理员账号、证明材料与 Logo 等可选附件）与申请状态查询；
    \item 管理端待审列表、检索、批准与拒绝组织申请；管理员直接创建组织等运维能力；
    \item 组织列表、详情、更新与删除，以及组织角色权限配置（与业务权限位协同）；
    \item 邀请码生成与消费、成员持码加入指定组织并绑定角色（如发布者、审核者）；
    \item 组织使用情况查询（剩余非 LLM 与 LLM 调用次数等），供用户端或管理端展示。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 组织申请表单字段、证明文件与 Logo 上传流；管理员审批意见与操作类型（通过、拒绝）；
    \item 组织维护表单、组织级权限配置值、配额调整参数；
    \item 邀请码、用户加入组织时的身份与角色选择。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 申请单状态、组织标识、组织公开信息及管理员关联结果；
    \item 组织成员列表或用户侧可见的组织名称、组织 ID 及配额摘要；
    \item 配额不足或无权操作组织资源时的业务错误信息，供检测任务等模块提前拦截。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{OrganizationApplication}：记录待审申请、提交时间、状态及处理人，关联证明材料与 Logo 等附件字段；
    \item \texttt{Organization}：正式组织实体，包含名称、邮箱、管理员用户一对一关联、描述、Logo、证明存档及组织级剩余调用次数等；
    \item \texttt{InvitationCode}：与组织、角色绑定的一次性邀请码及有效期、使用标记，用于受控扩员。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 申请方提交 \texttt{OrganizationApplication}，系统持久化并通知管理端进入待审队列；
    \item 管理员审核：拒绝则更新申请状态；批准则创建 \texttt{Organization}、建立管理员 \texttt{User} 与组织的关联，并视需要初始化配额；
    \item 组织管理员或系统生成 \texttt{InvitationCode}，新用户凭码注册或绑定时写入用户的组织外键与角色；
    \item 检测任务执行前后，由业务层调用组织配额校验与扣减（可与用户个人配额策略组合，以当前实现为准）；
    \item 组织信息变更或删除时，级联或限制处理成员与历史资源引用，避免产生孤儿数据。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 对外提供组织申请、审批、组织 CRUD、邀请码与组织使用情况等 REST 接口，供用户端与管理端调用；
    \item 依赖用户与权限模块：审批人、组织管理员及成员均以 \texttt{User} 为主体，认证鉴权由用户模块完成；
    \item 与检测任务管理模块、资源管理模块协作：在创建任务或上传资源时校验组织成员身份与组织配额；
    \item 与通知消息模块协作：申请结果、邀请相关事件可向用户推送站内通知或邮件（若已接入）。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 组织名称、联系邮箱等应保证唯一性或明确冲突处理策略，避免审批通过后无法落库；
    \item 申请中暂存的管理员密码等敏感信息需加密存储，审批拒绝后应及时清理或脱敏；
    \item 配额扣减与回滚应与任务状态机一致，防止任务失败仍扣次数或重复扣减；
    \item 删除组织前须评估成员、任务与文件的引用关系，采用软删除或迁移策略以保证学术资源可追溯；
    \item 组织级权限与用户级权限并存时，应文档化优先级（例如检测接口以组织配额为准还是以用户为准），减少前后端理解偏差。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 支持多级组织、子部门与数据隔离域，满足大学院系或出版集团结构；
    \item 配额模型扩展为计费套餐、周期账单与弹性扩容，并与统计报表打通。
\end{itemize}

\subsection{资源管理模块（SD-ZYGL-0001）}

\subsubsection{模块目标}
本模块旨在建立学术资源的统一管理机制，实现对图像、论文（PDF）及评审文本（Review）的标准化存储、索引与预处理，为系统各鉴伪功能提供一致的数据访问接口。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 统一接入：处理各类学术资源的上传、校验与格式识别；
    \item 物理存储：负责文件在存储介质（本地或云端）的持久化；
    \item 元数据维护：记录资源指纹（MD5）、类型、归属人及引用关系；
    \item 内容解析：执行 PDF 图像提取、文本转码等特定预处理操作。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 资源上传与校验：支持分片上传，验证文件完整性及安全性；
    \item 自动化解析引擎：针对 Paper 类型资源，自动剥离 PDF 内嵌图像；
    \item 资源生命周期管理：提供资源的逻辑删除、过期清理及检索功能。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 原始文件：用户上传的学术资源文件（如图像、文本等）；
    \item 资源类别；
    \item 所有者。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 唯一标识ID（ResourceID）：为每个资源分配的唯一标识符，用于后续检索和管理。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{AcademicResource (实体)}：包含资源 ID、类型编码、存储路径、文件哈希值、父级资源 ID（针对提取出的子图）；
    \item \texttt{StorageConfig (配置)}：定义不同类型资源的存储策略（如存储位置、备份频率）。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 模块接收到资源请求后，首先通过指纹校验判断是否为重复资源；
    \item 若为新资源，则进行物理写入并同步至数据库；
    \item 针对PDF论文，系统会异步触发解析任务，将其中的图片提取并重新注册为关联资源，最后返回处理结果。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 与用户与权限模块的交互：在资源上传、下载或删除时，本模块需调用用户与权限模块验证当前操作者的身份（如Token/UID），确保资源的私密性与合规性；
    \item 与检测任务管理模块的交互：这是最核心的交互。资源管理模块为检测任务管理模块提供唯一标识ResourceID的查询接口。当鉴伪任务启动时，检测任务管理模块通过此接口获取文件的物理路径或内存流，从而将数据传递给算法检测层；
    \item 与图像、论文以及评审检测模块的交互：本模块为算法检测模块提供预处理后的标准化输入。例如，将PDF论文中提取出的图片序列化后，传递给图像检测模块。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 高性能存储：采用哈希目录结构存储文件，避免单目录下文件过多导致的性能下降；
    \item 解耦设计：资源处理逻辑与具体的AI算法解耦，确保资源模块的通用性。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 资源指纹去重：引入感知哈希（pHash）技术，识别视觉相似的重复图片；
    \item 多存储后端支持：未来可以考虑计划接入阿里云或腾讯云提高可用性。
\end{itemize}

\subsection{检测任务管理模块（SD-JCRW-0001）}

\subsubsection{模块目标}
实现鉴伪任务的全生命周期管理，负责任务的创建、调度、状态追踪及结果汇总，确保 AI 算法能高效、有序地处理用户提交的学术资源。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 任务编排：根据用户选择的检测类型（图像查重/论文查重/评审分析），匹配对应的检测模块；
    \item 异步调度：管理检测队列，支持高并发下的任务排队与分发；
    \item 状态监控：实时记录任务进度（等待中、检测中、已完成、失败）。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 任务创建中心：接收 ResourceID 和检测参数，校验任务合法性并入库；
    \item 调度引擎：基于消息队列（如 Redis/RabbitMQ）将任务分发至空闲的算法节点；
    \item 通过WebSocket或轮询接口向前端同步实时检测进度。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 资源标识（ResourceID）；
    \item 检测模式；
    \item 优先级。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 任务编号（TaskID）；
    \item 任务状态（如“等待中”、“检测中”、“已完成”、“失败”）；。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{DetectionTask (实体)}：包含TaskID、关联ResourceID、当前状态、开始/结束时间、耗时、异常信息；
    \item \texttt{AlgorithmRegistry (配置)}：记录系统中可用的算法插件列表及其参数约束。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 用户在界面发起检测请求后，该模块首先调用 资源管理模块 确认文件就绪；
    \item 随后在数据库创建初始化状态的任务记录，任务进入待处理队列，由调度引擎根据算法负载情况取出并执行；
    \item 算法执行完成后，模块更新任务状态为“已完成”，并触发报告服务模块的预处理。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 与资源管理模块交互：通过ResourceID获取物理文件路径，确保检测对象存在；
    \item 与图像、论文以及评审检测模块交互：作为算法的“雇主”，通过接口调用具体的推理服务并接收回调；
    \item 与报告服务模块交互：任务完成后，将TaskID传递给报告服务模块，由其生成可视化报告。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 容错处理：若算法层超时或崩溃，模块需具备自动重试（Retry）机制或超时熔断逻辑；
    \item 解耦设计：采用“任务-算法”解耦架构，新增算法插件时无需修改任务管理的核心逻辑。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 分布式调度：支持跨服务器的算法节点调度，应对大规模学术会议的检测峰值；
    \item 优先级体系：引入付费用户或紧急任务的优先处理通道。
\end{itemize}

\subsection{图像检测模块（SD-TPJC-0001）}

\subsubsection{模块目标}
本模块面向用户上传或系统解析得到的学术图像，完成自动化 AI 鉴伪、异常痕迹识别和可疑区域标注。该模块需要在现有图像检测能力基础上，为检测任务、结果展示、综合报告和人工审核提供结构化图像检测结果。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 接收资源管理模块登记的图像文件或从论文、压缩包、PDF 中提取出的图像资源；
    \item 调用图像检测模型或图像鉴伪算法，判断图像是否存在 AI 生成、复制伪造、拼接篡改、背景掩盖等风险；
    \item 记录图像检测的整体结论、置信度、子方法结果、可疑区域掩膜和可视化标注结果；
    \item 将图像检测结果回写到检测任务，并向报告服务模块、人工审核模块和通知消息模块提供结果数据。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 图像来源解析：对直接上传图像和从 PDF、压缩包、论文文档中抽取出的图像进行统一登记；
    \item 学术图像 AI 鉴伪：对图像进行整体风险判断，输出是否可疑、置信度和风险等级；
    \item 子方法检测结果汇总：保存多种检测方法的概率、掩膜矩阵和可视化结果；
    \item 图像可疑区域标注：在原图或标注图中展示可疑区域、区域编号、风险类型和局部说明。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 图像资源文件及其元数据，包括文件名称、文件类型、上传用户、所属组织、来源页码或来源文件；
    \item 检测任务信息，包括任务编号、任务状态、检测参数、是否启用大语言模型辅助判断等配置；
    \item 图像检测服务所需的模型参数、图像预处理参数和可视化标注参数。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 图像整体检测结果，包括是否可疑、置信度、检测时间、任务状态和文字判断说明；
    \item 子检测结果，包括各检测方法概率、掩膜图、掩膜矩阵、ELA 图像、LLM 可视化图像等解释性数据；
    \item 面向用户端、管理端和报告服务的结构化结果数据。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{FileManagement}：记录用户上传文件的名称、大小、类型、标签和组织归属；
    \item \texttt{ImageUpload}：记录图像文件、来源文件、来源页码、检测状态、审核状态和图像级结论；
    \item \texttt{DetectionTask}：记录一次检测任务的发起用户、任务状态、检测参数和报告文件；
    \item \texttt{DetectionResult}：记录图像整体检测结论、置信度、检测状态、可视化结果和人工审核关联；
    \item \texttt{SubDetectionResult}：记录不同子检测方法的概率、掩膜图片和掩膜矩阵。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 用户上传图像或包含图像的文件后，资源管理模块完成文件校验、存储和图像资源登记；
    \item 检测任务管理模块创建图像检测任务，并将任务状态置为待处理或进行中；
    \item 图像检测模块读取图像资源，执行尺寸归一化、色彩空间转换、格式转换等预处理；
    \item 系统调用图像检测模型或外部 AI 服务，生成整体风险判断、子方法概率和可疑区域信息；
    \item 检测结果写入 \texttt{DetectionResult} 与 \texttt{SubDetectionResult}，同步更新图像和任务状态；
    \item 报告服务模块读取图像检测结果生成报告，通知消息模块向用户推送任务完成状态。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 与资源管理模块交互，获取已上传图像、PDF 提取图像或压缩包解析图像；
    \item 与检测任务管理模块交互，接收任务调度请求并回写任务状态；
    \item 与 AI 推理服务交互，提交图像文件和检测参数，接收检测概率、掩膜和可视化结果；
    \item 与报告服务模块交互，提供图像检测结论、可疑区域和解释性数据；
    \item 与人工审核模块交互，为用户提交人工审核申请和审核人员复核提供图像检测依据。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 检测过程应采用异步任务方式执行，避免大图、批量图像和外部模型调用阻塞用户请求；
    \item 图像资源、检测任务、检测结果和报告之间应保持可追溯关系，便于后续复核和问题排查；
    \item 子检测方法结果应以可扩展结构保存，支持后续新增检测方法或替换模型；
    \item 可疑区域标注应与原始图像保持坐标一致，避免前端展示、报告生成和人工审核之间出现位置错位；
    \item 模型调用失败、图像损坏、格式不支持等异常应回写明确状态，避免任务长期停留在进行中。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 扩展图像检测模型适配器，使系统支持不同鉴伪模型、不同检测策略和模型版本切换；
    \item 将图像检测结果纳入统一资源与综合鉴伪结果模型，便于与论文检测和评审检测结果合并生成综合报告；
    \item 增强图像局部解释能力，支持像素级高亮、区域级风险说明和人工复核标注回写。
\end{itemize}

\subsection{论文检测模块（SD-LWJC-0001）}

\subsubsection{模块目标}
本模块面向用户上传的全篇论文，完成论文文本解析、AIGC 风险检测、段落级概率分析和事实性鉴伪说明。该模块是本轮迭代在现有图像检测基础上扩展出的文本检测能力，设计上应接入统一检测任务和报告生成链路。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 对论文文件或论文文本进行格式校验、文本抽取、结构化解析和段落切分；
    \item 调用论文 AIGC 检测模型，输出全文级 AI 贡献占比、整体风险结论和结果摘要；
    \item 生成段落级 AI 生成概率、疑似 AI 生成段落列表、事实性鉴伪子结论及原因说明；
    \item 将论文检测结果写入统一检测任务，并为结果展示、综合报告和人工审核提供数据。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 全篇论文 AIGC 检测：输出全文级检测结论、AI 贡献占比或等价风险指标；
    \item AI 生成段落识别：标识疑似 AI 生成段落及其段落编号、文本内容和原文位置；
    \item 段落级 AI 生成概率分析：为段落生成概率、风险分值或风险等级；
    \item 事实性鉴伪子结论生成：针对论文中的可疑内容形成条目化子结论；
    \item 事实性鉴伪原因说明：为子结论提供原因、依据摘要和关联段落说明。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 用户上传的论文文件或论文文本，包含 PDF、DOCX、TXT 等系统支持格式；
    \item 论文资源元数据，包括标题、文件名、上传用户、所属组织、学科标签和任务编号；
    \item 论文检测参数，包括是否启用事实性鉴伪、段落切分策略、模型版本和阈值配置。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 论文全文级检测结果，包括整体风险等级、AI 贡献占比、结果摘要和检测状态；
    \item 段落级检测结果，包括段落编号、段落内容、AI 生成概率、风险标签和原文定位信息；
    \item 事实性鉴伪结果，包括子结论、原因说明、证据摘要和关联段落；
    \item 可被综合报告、用户端详情页和人工审核模块复用的结构化结果数据。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{FileManagement}：复用现有文件管理对象记录论文文件的上传信息和归属关系；
    \item \texttt{DetectionTask}：复用统一检测任务对象记录论文检测任务的状态、发起人和报告文件；
    \item 论文资源对象：建议在后续数据设计中新增或扩展，用于记录论文标题、正文文本、解析状态和结构化段落；
    \item 论文检测结果对象：建议用于记录全文级风险、AI 贡献占比、检测摘要和模型版本；
    \item 段落检测结果对象：建议用于记录段落编号、段落内容、段落级概率、风险标签、事实性子结论和原因说明。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 用户上传论文文件或提交论文文本后，资源管理模块完成格式校验、文件存储和资源登记；
    \item 检测任务管理模块创建论文检测任务，并记录任务类型、检测参数和初始状态；
    \item 论文检测模块解析论文文本，完成正文抽取、章节识别、段落切分和清洗；
    \item 系统调用论文 AIGC 检测模型，生成全文级风险结论和段落级概率结果；
    \item 系统基于高风险段落生成事实性鉴伪子结论和原因说明，并建立结果与原文段落的映射关系；
    \item 检测结果写入统一任务结果，并触发报告生成、结果展示和通知推送。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 与资源管理模块交互，接收论文文件、解析后的文本内容和论文元数据；
    \item 与检测任务管理模块交互，接收任务调度请求并更新论文检测状态；
    \item 与 AI 推理服务或文本检测适配器交互，提交论文文本、段落列表和检测参数；
    \item 与报告服务模块交互，提供全文结论、段落级结果、事实性子结论和原因说明；
    \item 与人工审核模块交互，为论文人工复核提供疑似段落、概率结果和事实性鉴伪依据。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 论文解析与模型检测应分阶段处理，解析失败、检测失败和结果生成失败应分别记录状态；
    \item 段落编号、段落内容、概率结果和事实性说明之间应保持稳定映射，避免前端高亮和报告内容错位；
    \item 论文文本可能包含公式、表格、参考文献和附录，检测前应明确正文抽取和噪声过滤策略；
    \item 论文内容属于敏感学术资源，结果展示、下载和人工审核流转应遵守用户权限和组织边界；
    \item 论文检测模型应通过适配器接入，避免后续替换模型时影响上传、任务和报告主流程。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 增加论文专用数据表或统一资源子类型，用于保存论文结构、段落索引和检测版本；
    \item 支持多模型融合检测，将 AIGC 检测、事实性鉴伪、引用异常检测等能力统一聚合；
    \item 支持论文检测结果与图像检测结果联动，形成面向整篇论文的综合鉴伪报告。
\end{itemize}

\subsection{评审检测模块（SD-PSJC-0001）}

\subsubsection{模块目标}
本模块面向同行评审 Review 文本，检测其是否存在 AI 生成、异常生成或模板化倾向。该模块用于补充论文与图像检测之外的评审文本风险识别能力，为虚假评审识别、综合鉴伪报告和人工审核提供依据。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 接收用户上传或输入的同行评审文本，并完成格式校验、文本抽取和预处理；
    \item 调用评审文本检测模型，判断 Review 是否存在 AI 生成倾向、异常文本特征或模板化表达；
    \item 输出风险等级、置信度、可疑语句、模板化短语和结构化说明；
    \item 将评审检测结果纳入统一检测任务，并支持后续报告生成、日志追踪和人工复核。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 同行评审 Review 检测：识别评审文本是否存在 AI 生成或异常生成倾向；
    \item 模板化评审倾向识别：识别固定句式、高重复表达、常见模板短语和异常长度特征；
    \item 可疑语句定位：返回可疑句子、段落位置、风险标签和解释说明；
    \item 评审检测结果归档：保存检测状态、风险评分、模型输出和报告所需结构化数据。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 用户上传的 Review 文本文件或在线输入内容，包含 TXT、DOCX、PDF 等系统支持格式；
    \item Review 元数据，包括上传用户、所属组织、任务编号、语言类型和可选的论文关联信息；
    \item 评审检测参数，包括 AI 生成阈值、模板短语库、相似度比对策略和模型版本。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item Review 整体检测结果，包括是否疑似 AI 生成、风险等级、置信度和结果摘要；
    \item 模板化倾向结果，包括模板化风险评分、高重复语句、模板短语匹配和相似文本提示；
    \item 可疑文本定位结果，包括句子或段落位置、风险标签和解释说明；
    \item 面向综合报告、用户详情页、管理端日志和人工审核的结构化结果数据。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{FileManagement}：复用现有文件管理对象记录 Review 文件的上传信息、类型和归属关系；
    \item \texttt{DetectionTask}：复用统一检测任务对象记录 Review 检测任务状态、用户和报告文件；
    \item Review 资源对象：建议在后续数据设计中新增或扩展，用于保存 Review 文本、语言类型、来源信息和解析状态；
    \item Review 检测结果对象：建议用于记录 AI 生成风险、模板化倾向、可疑语句、置信度和模型版本；
    \item 检测日志对象：用于记录评审检测过程、异常情况和管理端查询所需的追踪信息。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 用户上传 Review 文件或输入 Review 文本后，资源管理模块完成格式校验和资源登记；
    \item 检测任务管理模块创建 Review 检测任务，并记录任务类型、检测参数和初始状态；
    \item 评审检测模块对文本进行清洗、分句、分词、语言标准化和结构化预处理；
    \item 系统调用评审检测模型，计算 AI 生成概率、语义重复率、语言流畅度异常和模板化风险；
    \item 系统生成整体风险结论、模板化倾向说明和可疑语句定位结果；
    \item 检测结果写入统一任务结果，并触发报告生成、用户通知和日志记录。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 与资源管理模块交互，获取 Review 文件、文本内容和元数据；
    \item 与检测任务管理模块交互，接收任务调度请求并回写检测状态；
    \item 与 AI 推理服务或文本检测适配器交互，提交 Review 文本和模板化检测参数；
    \item 与报告服务模块交互，提供整体风险、模板化结果、可疑语句和解释说明；
    \item 与统计日志模块交互，记录 Review 检测任务执行过程、异常信息和管理端检索数据。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item Review 文本通常篇幅较短，检测设计应避免仅凭长度或固定阈值给出过度结论；
    \item 模板化检测应区分正常评审常用表达和异常批量生成表达，降低误判风险；
    \item 检测结果应提供可疑语句或模板化片段，不能只返回单一风险分数；
    \item 模板短语库、相似度比对策略和模型阈值应可配置，便于后续在管理端进行调整；
    \item Review 检测结果应与权限、组织边界和日志追踪机制保持一致，防止评审内容泄露。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 支持中英文及多语言 Review 检测，并根据语言类型选择不同模型或模板规则；
    \item 支持历史 Review 相似度比对，用于发现批量复制、模板化改写和虚假评审行为；
    \item 将 Review 检测结果与论文检测、人工审核意见合并，形成面向投稿或评审流程的综合风险判断。
\end{itemize}

\subsection{人工审核模块（SD-RGSH-0001）}

\subsubsection{模块目标}
本模块的目标是实现学术资源的人工审核功能，支持图像、论文和评审结果的人工审核。通过专家审核任务分配与管理，确保AI检测结果的准确性与可信度。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item \textbf{人工审核任务管理：} 分配和管理用户提交的人工审核请求，包括图像、论文和评审的审核。
    \item \textbf{专家审核任务分配：} 根据AI检测结果分配审核任务，确保任务按时完成并汇总结果。
    \item \textbf{审核结果汇总：} 汇总所有专家的审核结果，生成最终的人工审核报告。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item \textbf{功能点 1：任务分配：} 根据AI检测结果分配人工审核任务给相关领域的专家，保证任务的针对性与效率。
    \item \textbf{功能点 2：专家审核：} 专家依据AI检测结果进行人工审核，标注图像问题、分析论文内容或评审质量，最终提交审核报告。
    \item \textbf{功能点 3：审核报告生成：} 汇总专家的审核结果并生成详细的人工审核报告，提供最终结论。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item AI检测结果（图像/论文/评审）；
    \item 审核任务分配信息（专家任务分配、任务详情）。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 审核结果（人工审核报告）；
    \item 审核任务完成情况（专家评分、审核意见）。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \textbf{审核任务：} 记录审核任务的基本信息，包括任务类型、分配专家、审核状态等。
    \item \textbf{专家审核结果：} 包含专家的评分、审核意见、标注信息等。
    \item \textbf{人工审核报告：} 汇总所有专家审核意见的最终报告，包含检测结论、建议等。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 系统根据AI检测结果生成人工审核任务，并分配给相关专家；
    \item 专家接收任务并进行人工审核，标注问题并提供审核意见；
    \item 专家完成审核后提交结果，系统汇总所有专家的审核报告；
    \item 最终审核结果生成报告，供用户查看与下载。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item \textbf{任务分配接口：} 系统与任务分配平台交互，分配审核任务；
    \item \textbf{审核提交接口：} 专家通过系统提交审核结果，上传标注内容；
    \item \textbf{报告生成接口：} 将专家审核结果生成最终报告，并提供下载接口。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 确保任务分配的精确性和及时性，保证审核任务高效流转；
    \item 审核结果的可追溯性，确保每个专家的评分和评论能够追溯和验证；
    \item 设计合理的专家任务管理系统，支持审核进度跟踪。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
   \item 增强专家审核质量控制，提供更多维度的专家评价；
   \item 支持多语言或跨领域的专家审核。
\end{itemize}

\subsection{通知消息模块（SD-TZXX-0001）}

\subsubsection{模块目标}
本模块用于承载系统内与检测任务、人工审核、组织审批和管理操作相关的消息通知能力，保证用户能够及时获知任务状态变化、审核进展和系统事件，从而支撑平台的可用性与协作效率。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 统一接收业务模块产生的通知事件，并将其转换为站内消息记录；
    \item 为用户端和管理端提供通知列表、未读数量、标记已读与消息跳转能力；
    \item 通过 WebSocket 与轮询接口向前端同步关键业务状态变化，降低用户等待过程中的信息不透明问题。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 消息生成：根据检测完成、人工审核申请、组织审批结果、管理员操作等事件生成通知；
    \item 消息消费：提供通知列表查询、未读数统计、消息详情查看和已读标记能力；
    \item 实时推送：通过 WebSocket 向在线用户主动推送状态更新，离线场景下由前端通过接口补拉消息。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 来自检测任务、人工审核、组织审批、后台管理等模块的业务事件；
    \item 当前用户身份、目标接收者、消息类别、消息标题、消息正文和跳转链接等消息元数据。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 持久化后的通知记录、未读状态统计结果和消息查询结果；
    \item 面向前端的实时推送事件或标准 REST 响应，供页面更新通知中心与状态提示。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{Notification}：系统通知实体，记录收发双方、消息类别、标题、正文、已读状态、发送时间与跳转地址；
    \item 业务事件上下文：由其他模块传入的任务编号、审核请求编号、组织申请编号等引用信息，用于拼装通知内容；
    \item WebSocket 连接上下文：用于识别在线用户、定位推送目标并维护消息到达的实时性。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 业务模块在状态变化或关键操作完成后发出通知事件，并传入接收者、类别和业务上下文；
    \item 本模块根据事件类型组装通知标题、正文与跳转链接，生成 \texttt{Notification} 记录；
    \item 若接收者在线，则通过 WebSocket 向对应会话推送消息摘要；若接收者离线，则等待其下次登录后通过查询接口获取；
    \item 用户端或管理端读取通知中心数据，并可将消息标记为已读或跳转至对应业务页面。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 接收检测任务管理模块、人工审核模块、组织与成员模块和管理端治理接口发出的消息事件；
    \item 对前端提供通知列表、未读消息数量、消息状态更新和 WebSocket 推送接口；
    \item 与用户与权限模块协同，校验当前访问者身份，确保通知只能被对应接收者或管理员读取。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 通知内容应避免直接复制大段业务数据，正文只保留必要摘要，详细信息由跳转页面展示；
    \item 同一业务事件的重复触发需有幂等控制，避免因任务重试或状态重复回写产生通知风暴；
    \item 实时推送与持久化消息应解耦，推送失败不影响消息记录落库；
    \item 通知分类需要兼顾当前实现与后续扩展，至少支持系统消息、全局公告、发布者到审核者和审核者到发布者等类型。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 扩展邮件、短信或企业微信等多渠道通知，以适配组织级部署场景；
    \item 引入通知模板中心与用户通知偏好配置，实现按事件类型的订阅与静默策略。
\end{itemize}

\subsection{统计日志模块（SD-TJRZ-0001）}

\subsubsection{模块目标}
本模块用于记录系统关键操作、检测过程和人工审核过程中的审计信息，并向管理端提供统计分析能力，为运行监控、问题追踪和后续治理决策提供数据支撑。

\subsubsection{模块职责}
\begin{itemize}[leftmargin=2em]
    \item 统一记录上传、检测、人工审核、组织审批和后台治理等关键操作日志；
    \item 对任务数量、检测结果、审核分布、用户活跃度等业务指标进行汇总统计；
    \item 为管理员提供可检索、可筛选、可追溯的日志与统计数据访问入口。
\end{itemize}

\subsubsection{功能组成}
\begin{itemize}[leftmargin=2em]
    \item 日志采集：采集用户上传、检测提交、审核处理、组织审批、管理操作等事件；
    \item 统计分析：对用户数、检测数、审核数、检测结论分布和时间趋势进行聚合计算；
    \item 管理查询：支持管理端按时间、用户、组织、操作类型和业务对象进行筛选查询。
\end{itemize}

\subsubsection{输入与输出}
输入：
\begin{itemize}[leftmargin=2em]
    \item 各业务模块产生的操作事件、状态变更事件和业务主键引用；
    \item 管理端提交的统计维度、时间范围、筛选条件和排序参数。
\end{itemize}

输出：
\begin{itemize}[leftmargin=2em]
    \item 持久化日志记录、审计轨迹和按条件检索后的日志列表；
    \item 面向管理端仪表盘的统计汇总结果，如趋势数据、分布数据和总量数据。
\end{itemize}

\subsubsection{核心数据对象}
\begin{itemize}[leftmargin=2em]
    \item \texttt{Log}：操作日志实体，记录操作时间、用户、操作类型、关联模型名称和关联对象编号；
    \item 业务统计聚合结果：由检测任务、检测结果、人工审核、用户与组织等实体汇总得到的统计数据；
    \item 管理端查询条件对象：封装筛选范围、排序方式、分页参数和维度组合。
\end{itemize}

\subsubsection{处理流程}
\begin{enumerate}[leftmargin=2em]
    \item 用户或管理员完成上传、检测、审核、组织审批等操作后，业务模块生成对应日志事件；
    \item 本模块将事件映射为统一的 \texttt{Log} 记录并写入数据库，必要时补充关联对象信息；
    \item 管理端根据时间、组织、用户或事件类型查询日志或调用统计接口；
    \item 本模块执行聚合分析后返回仪表盘数据、列表数据或导出所需的数据集。
\end{enumerate}

\subsubsection{模块接口与交互关系}
\begin{itemize}[leftmargin=2em]
    \item 接收用户与权限、组织管理、资源管理、检测任务、人工审核和通知模块写入的操作事件；
    \item 向管理端提供日志查询接口、统计接口和数据概览接口；
    \item 与报告服务模块和后续运维模块协同，为阶段性汇总报告和系统运行分析提供原始数据。
\end{itemize}

\subsubsection{设计要点}
\begin{itemize}[leftmargin=2em]
    \item 日志字段设计应突出“谁在何时对什么做了什么”，优先保证可追溯性而不是过度冗余；
    \item 高频统计查询应与原始日志写入解耦，避免管理端分析请求影响主业务接口性能；
    \item 统计口径需在文档中固定，如检测次数按任务计还是按图片计，以免前后端和报告口径不一致；
    \item 涉及个人信息和敏感业务数据时，日志展示需做脱敏或按角色限制访问范围。
\end{itemize}

\subsubsection{后续扩展方向}
\begin{itemize}[leftmargin=2em]
    \item 增加按组织、学科标签、模型版本和检测类型的多维统计分析能力；
    \item 支持日志归档、离线分析和异常行为告警，为后续运维监控与审计合规提供基础。
\end{itemize}

\subsection{本章小结}
本章在总体架构基础上，将系统划分为用户与权限、组织与成员、资源管理、检测任务、图像检测、论文检测、评审检测、人工审核、报告服务、通知消息和统计日志等核心模块，并统一说明了各模块的职责边界、输入输出、核心对象和协作关系。通过该划分，当前项目中已经落地的图像检测能力可以继续稳定运行，而论文检测、评审检测、统一报告和治理统计等迭代需求也具备了清晰的落点。

后续详细设计阶段，可在本章模块边界基础上继续细化数据库表结构、接口参数、状态机和异常处理策略，并按模块逐步推进开发与联调，以降低后续迭代时的耦合风险。
