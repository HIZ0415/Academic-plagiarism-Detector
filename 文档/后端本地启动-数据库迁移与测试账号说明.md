## 1. 这次核对后的结论

当前项目的**模型、迁移文件、sqlite 表结构是对齐的**。

已确认：

- `core` 当前只有一份正式迁移：`core/migrations/0001_initial.py`
- `DetectionTask` 的 `task_type`、`paper_file`、`error_message` 以及相关索引已经在 `0001_initial` 中
- `manage.py makemigrations --check --dry-run core` 返回 `No changes detected`
- 本地开发实际使用的数据库文件是：
  - `代码/后端/db.sqlite3`

本次发现的主要问题不是“模型和表结构不一致”，而是下面这些**本地环境漂移点**：

1. 旧文档把 sqlite 路径写成了 `代码/后端/后端代码/db.sqlite3`
2. 如果没有显式指定 `DJANGO_SETTINGS_MODULE=fake_image_detector.local_settings`，`manage.py` 默认会连远端 MySQL
3. 一些同学可能沿用旧 sqlite 文件，导致本地测试数据、账号密码、样例任务数量都不同
4. 后端测试脚本里曾使用过旧密码 `Test123456`，现已统一修正为 `Publisher123!`

## 3. 团队统一的本地数据库对齐方案

### 3.1 如果你只想继续使用现有 sqlite

在后端目录执行：

```powershell
cd D:\github_projects\Academic-plagiarism-Detector\代码\后端\后端代码
$env:DJANGO_SETTINGS_MODULE="fake_image_detector.local_settings"
$env:AI_SERVICE_URL="http://127.0.0.1:8010"
$env:AI_SERVICE_TIMEOUT_SECONDS="120"
& ".\.venv\Scripts\python.exe" manage.py migrate
& ".\.venv\Scripts\python.exe" scripts/init_test_users.py
```

适用场景：

- 你不想清空自己本地已有测试数据
- 你只想把数据库结构和测试账号校正到当前仓库标准

### 3.2 如果你想和组员得到**完全一致**的本地基线

推荐直接重建 sqlite：

```powershell
cd D:\github_projects\Academic-plagiarism-Detector\代码\后端\后端代码
$env:DJANGO_SETTINGS_MODULE="fake_image_detector.local_settings"
$env:AI_SERVICE_URL="http://127.0.0.1:8010"
$env:AI_SERVICE_TIMEOUT_SECONDS="120"

Remove-Item ..\db.sqlite3 -ErrorAction SilentlyContinue
& ".\.venv\Scripts\python.exe" manage.py migrate
& ".\.venv\Scripts\python.exe" scripts/init_test_users.py
```

这样会得到一份新的本地 sqlite，避免历史测试数据干扰。

注意：

- `Remove-Item ..\db.sqlite3` 删除的是 `代码/后端/db.sqlite3`
- 不是 `代码/后端/后端代码/db.sqlite3`

## 4. 哪些命令是“常规必跑”，哪些只是“旧库补救”

### 4.1 常规必跑

每次本地联调，推荐至少执行：

```powershell
cd D:\github_projects\Academic-plagiarism-Detector\代码\后端\后端代码
$env:DJANGO_SETTINGS_MODULE="fake_image_detector.local_settings"
$env:AI_SERVICE_URL="http://127.0.0.1:8010"

& ".\.venv\Scripts\python.exe" manage.py runserver 127.0.0.1:8000
```

### 4.2 仅旧库补救时使用

以下两个命令**不是新库必跑项**：

```powershell
& ".\.venv\Scripts\python.exe" manage.py migrate_detection_task_v2
& ".\.venv\Scripts\python.exe" manage.py backfill_detection_task_v2
```

原因：

- `DetectionTask v2` 字段已经进入 `0001_initial`
- 对于新建 sqlite，`manage.py migrate` 就能直接建出完整结构
- 这两个命令主要用于**历史数据库补字段 / 回填旧数据**

建议：

- 新同学首次搭环境：**不用默认执行这两个命令**
- 只有当你手上是很早以前的旧 sqlite，且怀疑 `DetectionTask` 缺字段时，再执行它们

## 7. 如何快速检查自己本地是不是“跑偏了”

### 7.1 检查是否误连了远端 MySQL

如果你没有先设置：

```powershell
$env:DJANGO_SETTINGS_MODULE="fake_image_detector.local_settings"
```

那么很多 `manage.py` 命令会默认走 `fake_image_detector.settings`，也就是远端 MySQL。

所以本地联调前，先确认：

```powershell
echo $env:DJANGO_SETTINGS_MODULE
```

应当看到：

```powershell
fake_image_detector.local_settings
```

### 7.2 检查迁移是否对齐

```powershell
& ".\.venv\Scripts\python.exe" manage.py showmigrations
& ".\.venv\Scripts\python.exe" manage.py makemigrations --check --dry-run core
```

预期：

- `core [X] 0001_initial`
- `No changes detected in app 'core'`

### 7.3 检查测试账号是否统一

```powershell
& ".\.venv\Scripts\python.exe" scripts/init_test_users.py
```

看到类似输出即可：

```powershell
synced organization: Local Demo Organization / local-org@example.com
updated user: admin@mail.com / Admin123! / role=admin
updated user: publisher_test@example.com / Publisher123! / role=publisher
updated user: reviewer_test@example.com / Reviewer123! / role=reviewer
```
