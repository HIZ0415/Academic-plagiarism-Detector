# 前后端对接说明（前端改造总览）

本文用于给后端同学快速接手：说明前端相较“最初图片鉴伪平台”做了哪些改造、当前代码已落地点、仍需后端配合的接口契约与联调优先级。

### 协作约定（重要）

- **用户端前端改造默认不修改后端仓库代码。** 涉及审稿接口字段扩展、`get_review_detail` 契约、登录/注册与 JWT 行为等，均由 **后端在服务端落地**。  
- **用户端「专家」（审稿人）与人工审核相关后端事项**（审稿列表/详情接口 + 登录注册建议）集中写在 **[`前后端对接-用户端专家模式.md`](./前后端对接-用户端专家模式.md)**（§4 接口契约、§6～§7 后端工作清单）。  
- **管理端专项** 仍以 **[`前后端对接-管理端.md`](./前后端对接-管理端.md)** 为准。

## 1. 从旧平台到当前版本：核心变化

最初平台以“图像鉴伪”为单一主线。当前前端已扩展为“统一学术检测”入口，覆盖三类资源与统一任务详情展示。

- 入口从单一图片上传，升级为统一上传（图像 / 论文 / Review 文本）。
- 检测任务从单链路，升级为多类型任务（图片检测、论文 AIGC、学术资源检测、Review 检测）。
- 增加批量提交、任务状态同步、统一结果详情视图。
- 增加人工审核流程可视化与结果展示页（当前可 mock 展示，不依赖管理端审批）。
- 管理端接口已完成类型化封装（用户、资源、日志、统计），并新增任务列表、统计摘要卡、登录页注册等；**管理端专项、需后端配合且约定「前端不改后端代码」的条目** 统一写在 **[`前后端对接-管理端.md`](./前后端对接-管理端.md)**，请后端同学以该文档为准在 `views_admin.py` 等处落地。
- **Alpha 任务 034 / 035**：**034** 本地启动文档与脚本完整联调 → **`文档/本地启动与停止手册.md`** §2（`scripts/start-local-windows.ps1` 等）；**035** 异步任务可观测（轮询或 WebSocket）→ **[`前后端对接-管理端.md`](./前后端对接-管理端.md) §5**（管理端 `/tasks` 实现轮询 + `ws/notifications`）。
- **用户端「专家」= 审稿人角色（人工审核）**：需求 **FR-YHZS-0007** 仅规定检测模式为 **快速 / 精准**；登录「专家」对应 **reviewer**。审稿人任务池 **`/review`**、详情 **`/task/detail/:manual_review_id`**、列表/详情接口字段与 **`get_review_detail`** 契约见 **[`前后端对接-用户端专家模式.md`](./前后端对接-用户端专家模式.md)**（已与 **管理端审批 `status2`**、发布者申请链路对齐）。

## 4. 后端对接重点（优先级）

### 4.1 用户端：审稿人与登录注册（前端不改后端）

以下为 **后端** 工作摘要；完整字段与行为说明见 **[`前后端对接-用户端专家模式.md`](./前后端对接-用户端专家模式.md)**。

| 类别 | 后端落地要点 |
|------|----------------|
| 审稿任务列表 | `GET /get_reviewer_tasks/`：筛选参数、`review_request_id`、`review_request_status`、`admin_gate_status`、`task_kind` 等，见该文档 §4。 |
| 审稿任务详情 | `GET /get_review_detail/<manual_review_id>/`：返回 **`imgs`**、**`review_request`**、**`ai_detection_result`**、**`manual_review_status`** 等，见该文档 §4。 |
| 登录 / 注册 | 公开登录/注册路径与 **无效 Bearer** 的优先级、邮箱规范化认证、邀请码与 `role` 一致、`is_used`、头像字段安全等，见该文档 **§6**。 |
| 联调与环境 | CORS、`localadmin` 仅管理端等，见该文档 **§7**。 |

### 4.2 检测与任务链（通用）

#### P0（先保证可跑通）

1. `GET /user-tasks/` 返回稳定字段（至少）：
   - `task_id`
   - `status`（`pending`/`in_progress`/`completed`/`failed`）
   - `upload_time`
   - `completion_time`
   - 建议补 `task_type`（`image_detection` / `paper_aigc` / `resource_check` / `review_detection`）

2. `GET /paper/tasks/{taskId}/status/` 返回：
   - `status`
   - `error_message`（失败时）
   - 可选 `progress`

3. `GET /paper/aigc/{taskId}/result/` 返回：
   - `task_id`
   - `ai_contribution_ratio`
   - `summary`
   - `paragraphs[]`（`index`、`risk_score`、`risk_level`、`excerpt`）

4. `GET /paper/resource-check/{taskId}/result/` 返回：
   - `task_id`
   - `suspected_risk_count`
   - `summary`
   - `issues[]`（`reference_index`、`issue_type`、`detail`、`severity`）

#### P1（补齐体验）

1. 统一任务模型增加 `task_type`，让前端无需推断即可正确分流专项页。
2. 人工审核结果聚合接口（用于替换当前 mock 的人工审核结果页）。
3. 图像/论文/Review 三类型统一结果结构（便于统一详情页完全真实化）。
4. **管理端**：`GET /get_all_user_tasks/` 按 [`前后端对接-管理端.md`](./前后端对接-管理端.md) **§2.1** 扩展字段及查询优化（**由后端实现；前端不在仓库内改** `views_admin.py`）。

## 5. 前端当前兜底逻辑（请后端知晓）

为保证演示与开发推进，前端做了以下兜底：

- 当任务刚创建但后端历史接口未及时回写时，前端把任务写入 localStorage 并在历史页合并展示。
- 人工审核结果页当前可用 mock 数据演示流程，后续可直接替换成真实接口。
- `VITE_USE_MOCK_AIGC=true` 时，部分接口走前端 mock 响应。

这意味着：即使后端尚未全部完成，页面也能先演示流程；但联调时请以后端真实返回为准。

## 8. 文件预处理对接补充

后端已在进入 AI 前完成论文和专家人工 Review 的预处理，前端不需要自行解析 PDF 或 TXT。

- `POST /paper/upload/` 只提交 PDF 文件。成功响应包含 `paragraph_count`，表示后端已完成 PDF 文本提取、清洗和段落切分。
- `POST /review/submit/` 只提交在线文本或 `.txt` 文件。成功响应包含 `cleaned_text_length`，表示后端已完成编码标准化和文本清洗。
- 前端文件选择器应保持当前约束：论文仅 `.pdf`，Review 文件仅 `.txt`，不要放开 DOCX、TXT 论文或 PDF Review 等新增格式。
- 任务状态接口不需要新增轮询端口；预处理产物缺失时后端会将任务置为失败并返回错误信息。
