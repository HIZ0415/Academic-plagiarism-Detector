# 前端联调闭环操作手册

本文说明**模式甲（真实四服务）**本地联调步骤。用户端已移除界面预览与前端 Mock，仅连接本机 Django。  
环境变量、端口、脚本细节与《[本地启动与停止手册](./本地启动与停止手册.md)》**第 7.1 节**一致；接口契约见《[前后端对接-总览](./前后端对接-总览.md)》**第 3 节**。  
功能实现进度见《[需求与概要设计-实现查漏补缺](./需求与概要设计-实现查漏补缺.md)》；**逐项验收**见《[验收自检清单](./验收自检清单.md)》（按表反馈通过/失败即可）。  
**一键脚本** `scripts/start-local-windows.ps1` 为可选；**最稳妥是下面「分步启动」**，不依赖脚本、不受分支是否合并脚本影响。

## 模式甲：立刻联调（真实后端，推荐）

### 第 1 步：准备

1. 使用 **Windows + PowerShell**。
2. 若从未允许过脚本，执行一次（当前用户即可）：
  ```powershell
   Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
  ```
3. 下文命令已按你的本机仓库根目录写好，**可直接整段复制到 PowerShell**（若你本机根路径不同，请全文替换为你的路径）：
  - `**G:\大学\大三下\软工\Academic-plagiarism-Detector`**

---

### 第 2 步（推荐 · 最稳妥）：分步启动 — 完全绕过一键脚本

**不要**求先 `git pull`。

**依赖**：后端目录下已有 `**.venv`**（若从未建过，需在 `代码\后端\后端代码` 里先按《本地启动与停止手册》建 venv 并 `pip install -r requirements.local.txt`）；数据库至少执行过一次 `**migrate**`（否则登录会报错）。

**终端 1 — AI（做检测、联调闭环时必开）**

```powershell
cd "G:\大学\大三下\软工\Academic-plagiarism-Detector\代码\AI服务\AI服务器代码"
& "G:\大学\大三下\软工\Academic-plagiarism-Detector\代码\后端\后端代码\.venv\Scripts\python.exe" ai_http_service.py --host 127.0.0.1 --port 8010
```

**终端 2 — Django（必开）**

本仓库**不要**只用裸的 `python manage.py runserver`，否则易错用全局 Python、且缺 `local_settings` / AI 地址。

```powershell
cd "G:\大学\大三下\软工\Academic-plagiarism-Detector\代码\后端\后端代码"
.\.venv\Scripts\Activate.ps1
$env:DJANGO_SETTINGS_MODULE = "fake_image_detector.local_settings"
$env:AI_SERVICE_URL = "http://127.0.0.1:8010"
$env:AI_SERVICE_TIMEOUT_SECONDS = "120"
python manage.py runserver 127.0.0.1:8000
```

安装 **daphne** 后，`runserver` 会走 ASGI，控制台应出现类似 `Starting ASGI/Daphne version ...`，通知 WebSocket `/ws/notifications/` 才可用（未装时仅 HTTP 正常，WS 会 404）。若依赖里尚无 daphne：

```powershell
pip install "daphne>=4.0,<5.0"
```

然后**重启** Django。

**终端 3 — 用户端（必开）**

用户端已固定为**真实本地联调**（无界面预览、无前端 Mock）；请求经 Vite 代理转发到 Django。可选：在 `**代码\前端\前端用户端\.env`** 中设置 `**VITE_API_URL=http://127.0.0.1:8000**`（不设置则使用相对路径 `/api`，效果相同）。

```powershell
cd "G:\大学\大三下\软工\Academic-plagiarism-Detector\代码\前端\前端用户端"
npm run dev -- --host 127.0.0.1 --port 3000 --strictPort
```

浏览器打开 `**http://127.0.0.1:3000/login**`（无痕亦可），用本机种子账号登录（若从未造数，需先在终端 2 里跑过迁移 + 种子；与一键脚本等价时可参考《后端本地启动》里造数说明）。

**终端 4 — 管理端（走「管理端审批」闭环时必开）**

```powershell
cd "G:\大学\大三下\软工\Academic-plagiarism-Detector\代码\前端\前端管理端"
# 若无 .env：新建文件写入一行 VITE_API_URL=http://127.0.0.1:8000
npm run dev -- --host 127.0.0.1 --port 3001 --strictPort
```

**两终端极简版（仅想先看页面/登录，不做检测）**：可只开 **终端 2 + 终端 3**；但未起 AI 时，**图像/论文检测会失败**，完整闭环仍需按上表开满四终端。

### 第 2 步（可选）：一键脚本 `start-local-windows.ps1`

在仓库根目录执行：


**关于 `git pull`**：不是联调必做。拉远端前：**保存全部文件** → `git status` → 有未提交改动则 `**git stash`** 或 `**git commit**`，再拉取。

### 第 4 步：默认账号（与脚本一致，未改参数时照抄）


| 用途                        | 打开                      | 邮箱                           | 密码              |
| ------------------------- | ----------------------- | ---------------------------- | --------------- |
| 管理端·组织审批（人工审核通过/拒绝）       | `http://127.0.0.1:3001` | `org_admin@example.com`      | `OrgAdmin123!`  |
| 管理端·软件管理员（跨组织治理，不审批本组织申请） | `http://127.0.0.1:3001` | `admin@mail.com`             | `Admin123!`     |
| 用户端·编辑                    | `http://127.0.0.1:3000` | `publisher_test@example.com` | `Publisher123!` |
| 用户端·专家                    | `http://127.0.0.1:3000` | `reviewer_test@example.com`  | `Reviewer123!`  |


同一浏览器里一个账号只有一种角色；闭环里请**退出再登**，或**两个浏览器**分别保持编辑与专家。

### 第 5 步：闭环操作（按顺序做，不要跳步）

目标：**发布者发起人工审核 → 管理员通过 → 专家提交 → 发布者看汇总**。

**A. 发布者（编辑）**

1. 打开 `http://127.0.0.1:3000/login`，开关选 **「编辑」**，用 **[publisher_test@example.com](mailto:publisher_test@example.com)** 登录。
2. 侧栏进入 **统一学术检测**（`/upload`），上传并完成至少一次 **图像类**检测（联调最稳），在 **检测历史**（`/history`）找到该任务，确认状态为 **已完成**，记下 **任务 ID**（数字）。
3. 侧栏进入 **人工审核申请**（`/annual`）→ **发起人工审核申请**。
4. 表单填写：
  - **关联自动检测任务 ID**：上一步的任务 ID；
  - **任务类型**：选 **图像检测**（与 `image_detection` 一致）；
  - **申请理由**：不少于约 10 个字；
5. 提交。列表里应出现一条申请（待管理端处理或进行中，以后端为准）。

**B. 组织管理员**

1. 新开窗口（或另一浏览器）打开 `http://127.0.0.1:3001/login`，用 **[org_admin@example.com](mailto:org_admin@example.com)** / **OrgAdmin123!** 登录（组织管理员；勿用 `admin@mail.com`，该账号为软件管理员，侧栏无审批按钮）。
2. 侧栏进入 **人工审核审批**（`/reviews`），找到上一步的申请 → **详情** → **通过**。
  - 若提示组织内没有审稿人：先在环境里保证存在 **reviewer** 角色用户（种子已有 `reviewer_test`），再重试通过。

**C. 专家（审稿人）**

1. 回到用户端站点：退出登录。
2. 登录页开关选 **「专家」**，用 **[reviewer_test@example.com](mailto:reviewer_test@example.com)** / **Reviewer123!** 登录。
3. 侧栏进入 **人工审核**（`/review`）任务池：应出现刚通过的任务（列表有 **进入审核** 按钮）。若为空，请确认管理端用的是 **[org_admin@example.com](mailto:org_admin@example.com)** 审批且已通过。
4. 点 **进入审核** 进入 `**/task/detail/<manual_review_id>`**，按页面完成复核并 **提交结论**（真实联调调用 `POST /post_review/<manual_review_id>/`）。

**D. 再回到发布者看结果**

1. 退出，开关选 **「编辑」**，再用 **[publisher_test@example.com](mailto:publisher_test@example.com)** 登录。
2. 打开 `/annual` 可看进度；打开 `/history` 进入该任务详情，在 **专家已完成** 且接口正常时，**查看人工审核结果** 可点，进入 `/manual-review-result` 看汇总。

至此闭环结束。

### 第 6 步：停止

- **若用一键脚本起的进程**：
  ```powershell
  cd "G:\大学\大三下\软工\Academic-plagiarism-Detector"
  .\scripts\stop-local-windows.ps1
  ```
- **若用分步启动**：在各终端窗口对 AI / Django / 前端进程 **Ctrl+C** 即可。

## 常见问题

1. **路径含中文**：一般可直接 `cd`；若乱码可先 `chcp 65001`。
2. **页面像假数据**：确认已用种子账号登录且 Django 在 `8000` 运行；用户端须能访问 `http://127.0.0.1:3000/api/login/`（或 `.env` 中的 `VITE_API_URL`）。
3. `**ParserError` /「字符串缺少终止符」**：改用 **「第 2 步（推荐）：分步启动」**，或 **PowerShell 7** 跑脚本。`**git pull` 不是必做**；拉代码前**保存 + stash/commit**。
4. **自检四条命令全部「无法连接」**：四进程未起或端口错；先按 **「第 2 步（推荐）」** 起服务再自检。
5. **编辑能登录、专家提示 `Invalid credentials`**：登录接口用 **邮箱 + 密码** 认证（`EmailBackend`），与是否点「专家」无关；401 表示**库里没有该邮箱**或**密码不对**。若在 Django Admin **手加用户**，常见漏项：**未填 `email`（只写了 username）**、**未用「修改密码」保存**（密码未哈希）、**邮箱与 `reviewer_test@example.com` 不一致**。角色错时后端会返回 **400** 与 `Invalid role`，而不是 401。可在 `manage.py shell` 对审稿人账号执行 `set_password` 并设 `role='reviewer'` 后重试。
6. **统一学术检测里论文 PDF「成功 0、失败 2」**：**不是**「没部署云端」的问题，而是本机 **Django 已收到请求但流程失败**。常见原因：**① 专家账号无上传权**（见上）；若已是**编辑**仍失败，请看 **②③④**。**② 发布者账号未绑定组织**——后端 `has_permission` 要求 `organization` 非空，否则上传/提交一律 **403**；请在 Django Admin 给用户选组织，或与种子脚本一样关联 `Local Demo Organization`。**③ AI 服务未起或 `AI_SERVICE_URL` 未设置**——先起 `8010`，Django 环境变量 `AI_SERVICE_URL=http://127.0.0.1:8010`。**④ PDF 预处理失败**（加密/扫描件无法抽正文等）——返回 **400**，`detail` 有说明。用户端队列表现在会显示 **「失败原因」**列；也可 **F12 → Network** 看 `paper/upload/`、`paper/aigc/submit/` 的响应体。
