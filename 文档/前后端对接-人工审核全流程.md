# 前后端对接 · 人工审核全流程（发布者 ↔ 管理端 ↔ 专家）

本文约定 **编辑（publisher）发起人工审核申请** → **管理员审批** → **审稿人（reviewer）鉴定并提交** → **发布者查看汇总结果** 的完整闭环，并说明三端前端已实现内容及 **后端必须落地的接口与状态机**。  
共性环境与 JWT 行为见 **[`前后端对接-前端改造总览.md`](./前后端对接-前端改造总览.md)**。

---

## 1. 流程与状态（后端建模建议）

### 1.1 实体建议

| 实体 | 说明 |
|------|------|
| **ManualReviewRequest**（或沿用现有 `ReviewRequest`） | 发布者发起的「人工审核申请」一行记录；与自动检测任务 `detection_task_id`（及可选 `batch_session_id`）关联。 |
| **ManualReview**（专家任务） | 管理端「通过」后为该申请生成专家侧任务，对应前端专家列表里的 **`manual_review_id`**（可与申请单 1:1）。 |

### 1.2 发布者可见状态（聚合）

| `admin_state` | 含义 |
|---------------|------|
| `pending` | 待管理端审批 |
| `refused` | 管理端拒绝，`admin_reject_reason` 必填 |
| `accepted` | 已通过，已生成 `manual_review_id`，专家可见 |

| `manual_review_status`（仅 `accepted` 后有意义） | 含义 |
|--------------------------------------------------|------|
| `undo` | 专家未完成 |
| `completed` | 专家已 `POST /post_review/<manual_review_id>/` |

### 1.3 管理端列表 `state`（与现有 `reviews.vue` 对齐）

沿用 **`pending` / `refused` / `accepted`**（前端 chips 已适配）。审批接口 **`choice`**：`1` = 通过，`0` = 拒绝；拒绝时 **`reason`** 必填。

### 1.4 专家任务池字段（与 `get_reviewer_tasks` 对齐）

专家列表需包含：`manual_review_id`、`task_kind`（`image` / `paper` / `review` 或与前端过滤器一致的枚举）、`admin_gate_status`（`passed` 仅在管理通过后下发）、`review_request_status`、`publisher_username` 等（详见 **[`前后端对接-用户端专家模式.md`](./前后端对接-用户端专家模式.md)**）。

## 2. 前端已实现行为（摘要）

| 端 | 页面 / 模块 | 行为 |
|----|-------------|------|
| **用户端（发布者）** | `annual.vue` | **「发起人工审核申请」**对话框：`detection_task_id`、`task_type`、`reason`、`priority`、`batch_session_id`；提交 **`POST /manual-review-requests/`**（或 Mock）。列表仍走 **`GET /get_publisher_review_tasks/`**（或与下文统一的列表别名）。 |
| **用户端** | `annual.vue` · Mock | `VITE_USE_MOCK_MANUAL_REVIEW_WORKFLOW=true` 时展开 **「模拟管理端审批」**，用于无后端时在同一浏览器内推进到专家池（**不同 dev 端口之间 localStorage 不共享**，管理端 Mock 与用户端 Mock 数据默认不通）。 |
| **用户端** | `history.vue` | 检测详情拉 **`GET /manual-review-requests/by-detection-task/`**（或 Mock）刷新 **人工审核流程** 文案与步骤；**「查看人工审核结果」**仅在专家已完成且管理未拒绝时启用，并带 **`review_request_id`** 跳转。 |
| **用户端** | `manual-review-result.vue` | **`GET /manual-review-requests/<id>/publisher-summary/`**（或 Mock）渲染汇总表。 |
| **用户端（专家）** | `review.vue`、`task/detail/[manual_review_id].vue`、`reviewer.ts` | Mock 下 **`get_reviewer_tasks`**、**`get_review_detail`**、**`post_review`** 走共享模块 `代码/前端/shared/manualReviewWorkflowMock.ts`。 |
| **管理端** | `reviews.vue`、`review.ts` | 列表 **`GET /get_reviewRequest/all/`**；详情 **`GET /get_reviewRequest/<id>/`**；处理 **`POST /handle_reviewRequest/<id>/`**；展示 **检测任务 ID / 类型 / 理由摘要**（后端补字段后自动显示）。 |

## 3. 后端接口契约（建议路径 · 均为 `/api/` 前缀）

以下为基础契约；若 Django 已存在同名旧视图，请 **路由别名** 指向下列语义，并在本文档修订记录中注明映射。

### 3.1 `POST /manual-review-requests/`（发布者 · **发起申请**）

**认证**：Bearer JWT，`role=publisher`。

**Body（JSON）**：

```json
{
  "detection_task_id": "string，必填，对应自动检测任务 ID",
  "task_type": "image_detection | paper_aigc | resource_check | review_detection",
  "reason": "string，必填，建议最少 10 字",
  "priority": "normal | urgent，可选，默认 normal",
  "batch_session_id": "string，可选，与用户端统一检测批次一致"
}
```

**响应 201**：

```json
{
  "review_request_id": 10042,
  "status": "pending_admin",
  "message": "可选，人类可读提示"
}
```

**业务规则建议**：

- 同一 `detection_task_id` 下 **未结案**（管理未拒绝且专家未完成）时是否允许重复申请：产品决定；前端当前 **不禁止** 重复提交，后端可返回 **409**。
- 写入 `ManualReviewRequest`，`admin_state=pending`。

---

### 3.2 `GET /get_publisher_review_tasks/`（发布者 · 申请列表 · **已有**）

前端 **`annual.vue`** 继续调用；建议在每条记录中 **补充**：

| 字段 | 说明 |
|------|------|
| `detection_task_id` | 关联自动检测任务 |
| `task_type` | 与申请一致 |
| `admin_state` | `pending` / `refused` / `accepted`（可与列表 `status` 映射） |

列表现有 `status`、`progress`（如 `0/1`）请与 §1 状态机一致，便于进度条展示。

---

### 3.3 `GET /manual-review-requests/by-detection-task/`（发布者 · **按检测任务查申请**）

**Query**：`detection_task_id`（必填）。

**响应 200（找到）**：

```json
{
  "found": true,
  "review_request_id": 10042,
  "detection_task_id": "…",
  "task_type": "image_detection",
  "admin_state": "accepted",
  "manual_review_id": 50001,
  "manual_review_status": "completed",
  "admin_reject_reason": null,
  "created_at": "2026-05-02T12:00:00Z"
}
```

**响应 200（未找到）**：

```json
{ "found": false, "detection_task_id": "…" }
```

（若更偏好 **404**，请全局约定并在前端 `manualReviewWorkflow.ts` 中增加对 404 的解析；当前实现以 **catch 视为无数据** 兼容。）

---

### 3.4 `GET /manual-review-requests/<review_request_id>/publisher-summary/`（发布者 · **结果汇总**）

**认证**：publisher；仅能访问 **本人** 申请。

**响应 200**（字段与前端 `manual-review-result.vue` 一致即可）：

```json
{
  "review_request_id": 10042,
  "detection_task_id": "…",
  "task_type": "image_detection",
  "admin_state": "accepted",
  "manual_review_status": "completed",
  "manual_review_id": 50001,
  "summary": {
    "reviewerCount": 1,
    "suspiciousImageCount": 1,
    "finalDecision": "string"
  },
  "reviewerRows": [
    { "reviewer": "…", "decision": "…", "confidence": 85, "comment": "…" }
  ],
  "imageRows": [
    { "imageId": "…", "aiResult": "…", "manualResult": "…", "riskLevel": "…", "note": "…" }
  ]
}
```

论文 / Review 类无逐图时 **`imageRows`** 可为 `[]`。

---

### 3.5 管理端（与现有页面对齐）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/get_reviewRequest/all/` | 分页列表；建议每条含 **`detection_task_id`、`task_type`、`reason`**（全文或摘要）。 |
| GET | `/get_reviewRequest/<id>/` | 详情；建议返回 **`reason`、`detection_task_id`、`task_type`、`priority`、`imgs`（可选）、persons（分配专家占位）**。 |
| POST | `/handle_reviewRequest/<id>/` | Body：`{ "choice": 1 \| 0, "reason": "拒绝必填" }`。**通过**：创建 **`manual_review_id`**，写 **`admin_state=accepted`**，专家列表可见；**拒绝**：**`admin_state=refused`**。 |

---

### 3.6 专家端（与现有页面对齐）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/get_reviewer_tasks/` | 仅返回 **`admin_gate_status` 已通过** 的任务（Mock 与之一致）。 |
| GET | `/get_review_detail/<manual_review_id>/` | 材料单元、`task_kind`、`manual_review_status` 等。 |
| POST | `/post_review/<manual_review_id>/` | 专家提交结论；成功后 **`manual_review_status=completed`**，发布者可拉 §3.4。 |

## 4. 无后端联调（Mock）
