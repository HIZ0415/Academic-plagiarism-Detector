## 2. 概念（避免与「检测模式」混淆）

| 术语 | 含义 |
|------|------|
| **快速 / 精准** | 学术检测 **AI 策略**，仅两种（需求 FR-YHZS-0007）；**不是**第三种「专家检测」 |
| **登录「专家」** | 用户端展示文案；实现上 JWT **`role === 'reviewer'`** |
| **审稿人任务** | 人工审核流程中，**管理端已通过**后分配给审稿人的 **`manual_review_id` 任务** |

**不要实现**：在 `/upload` 增加与「快速」「精准」并列的第三种「专家」检测模式，或虚构 `detection_profile: expert` 等未立项字段。

---

## 3. 路由与信息架构

| 能力 | 路由 | 说明 |
|------|------|------|
| 任务池 | `/review` | 列表：筛选、分页、`task_kind`、门闸与流程状态 |
| 任务详情 | `/task/detail/<manual_review_id>` | 按 `task_kind` 展示图像逐图 / 论文片段 / Review 片段工作台 |
| 界面预览 | `preview_kind` query（详见代码） | 无 JWT 盲改 UI |

**守卫**：`reviewer` 访问发布者路径会被重定向到 `/review`（见 [`前后端对接-总览.md`](./前后端对接-总览.md) §2）。

**相关源码**：`src/pages/review.vue`、`src/pages/task/detail/[manual_review_id].vue`、`src/api/reviewer.ts`、`src/router/index.ts`。

## 4. 审稿接口契约（后端必须对齐）

以下路径相对于 **`/api`**（见 [`前后端对接-总览.md`](./前后端对接-总览.md) §1）。

### 4.1 `GET /get_reviewer_tasks/`

- **认证**：`role === 'reviewer'`。
- **Query（与前端列表对齐）**：`page`、`page_size`、`query`（发布者用户名前缀）、`status`（本人手工任务 `undo`/`completed`）、`start_time`、`end_time`、**`review_request_status`**、**`admin_gate_status`**、**`task_kind`**（筛选可选）。
- **每条记录建议包含**：
  - `manual_review_id`、`manual_review_time`
  - `publisher_username`、`publisher_avatar`
  - `status`（本人任务进度）
  - **`review_request_id`**、**`review_request_status`**、**`admin_gate_status`**
  - **`task_kind`**：前后端统一一套枚举（建议涵盖图像类、论文类、Review 类；可与 `image_detection`/`paper_aigc`/`review_detection` 映射）
  - **`image_count`**（或论文/Review 用段数、单元数，与列表「规模」列一致）
- **分页字段**：与现有前端解析一致，如 `results`、`current_page`、`total_pages`、`total_count`（若与现网字段名不同，需与前端 `review.vue` 对齐或后端加兼容层）。

**门闸**：仅当管理端对申请 **通过** 后，专家池才应出现该条；与 Mock 行为一致。

### 4.2 `GET /get_review_detail/<manual_review_id>/`

- **认证**：当前用户必须为该项 **`ManualReview.reviewer`**。
- **建议返回**：
  - **`task_kind`**：与列表枚举一致，驱动详情页工作台切换。
  - **图像类**：**`imgs`**：`[{ id, url }, ...]`，与 `POST /post_review/` 中 `img_id` 一致。
  - **论文 / Review 类**：在不使用 `imgs` 时提供 **`segments`** 或 **`text_units`**（结构化列表，字段名落地后若与前端占位不一致需一次性对齐）。
  - **`review_request`**：`id`、`status`（流程）、**`admin_gate_status`**、`request_time` 等。
  - **`ai_detection_result`**：摘要结构（图像可为置信度等；非图像可为摘要对象）。
  - **`manual_review_status`**：`undo` | `completed`。

### 4.3 `POST /post_review/<manual_review_id>/`

- **Body**：前端按任务类型组装（图像：`result[]` 含七维评分、造假判定、`img_id` 等；论文/Review：文本单元鉴定结论）。  
- **成功后**：`manual_review_status` → `completed`，发布者可拉取 [`前后端对接-总览.md`](./前后端对接-总览.md) §3.4 **publisher-summary**。

前端入口：`reviewer.submitReview(manual_review_id, data)` → `代码/前端/前端用户端/src/api/reviewer.ts`。

### 4.4 与其它接口的关系

- **管理端审批** 使用的 **`GET /get_reviewRequest/...`**、**`handle_reviewRequest`** 与用户端 **`get_review_detail`** 职责不同，**勿混用路径**。
- 论文自动检测 **`POST /paper/aigc/submit/`** 等仍为 **`paper_file_id`、`task_name`** 等既有字段，**不设**需求外的 `detection_profile`。

## 5. 后端工作清单：登录 / 注册 / JWT（审稿人与发布者共用）

| 项 | 建议行为 |
|----|----------|
| 公开路径与 Bearer | `POST /login/`、`POST /register/` **不应**因无效 `Authorization` 在校验账号密码**之前**直接 401 |
| 邮箱登录 | trim、大小写不敏感查询、`is_active` 校验 |
| `authenticate` | 传入 `request`，与中间件一致 |
| `AUTHENTICATION_BACKENDS` | 邮箱后端优先于通用 ModelBackend（若团队启用多后端） |
| 邀请码注册 | 角色以邀请码为准；请求 `role` 与邀请码不一致 → 400；核销 `is_used` |
| 登录响应 | `avatar` 缺失不得导致 500 |
