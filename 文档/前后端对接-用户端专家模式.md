# 前后端对接 · 用户端「专家」（审稿人）与人工审核

> **文档依据**：需求规格说明书（`文档/requirements`）中 **FR-YHZS-0007** 仅定义学术检测的两种可选模式：**快速检测模式**、**精准检测模式**，**不存在**名为「专家」的第三种检测模式或测评模式。概要设计 **质量属性设计**（`overview-design/sections/09-quality-design.tex`）亦未单独定义「专家检测模式」。  
> **「专家」在用户端的正确含义**：与登录身份相关的 **审稿人（reviewer）** 能力，服务于 **人工审核 / 专家审核** 流程（需求 **用户审核模块** `03-4-user-review.tex`；概要设计 **人工审核**、`04-module-design.tex` 中专家审核任务分配等表述）。

本文说明：**如何从产品语义区分「检测模式」与「专家角色」**，以及审稿人侧对接时前端涉及的能力与接口索引（具体路径以前端代码与 **[`前后端对接-前端改造总览.md`](./前后端对接-前端改造总览.md)** 为准）。

### 协作约定（重要）

- **用户端前端迭代默认不修改后端仓库代码。**  
- 本文 **§4～§6** 所列接口字段、认证与注册行为，均由 **后端同学在服务端自行落地**（实现文件与分支由后端团队安排）。  
- 前端仅以本文档为契约进行联调；若后端尚未实现某字段，前后端需另行约定兼容策略。

## 1. 概念对照（避免混淆）

| 术语 | 含义 | 文档出处（摘要） |
|------|------|------------------|
| **快速 / 精准检测模式** | 用户在学术检测流程中对 **AI 自动检测** 的策略选择，仅两种 | FR-YHZS-0007（`03-2-user-display.tex`） |
| **登录「专家」/ 审稿人** | 具备处理 **人工审核任务** 的用户角色，查看 AI 报告、提交鉴定结论与理由等 | `03-4-user-review.tex`；概要设计人工审核流程 |
| **专家审核任务** | 系统根据 AI 检测与流程规则分配的 **人工复核** 任务，由审稿人等执行 | 概要设计 `04-module-design.tex` 人工审核模块 |

**不应实现**：在 **`/upload`** 等检测入口增加与「快速」「精准」并列的第三种「专家检测模式」，或虚构 `detection_profile: expert` 等需求未约定的字段——除非将来需求变更并以书面需求为准。

## 2. 如何在前端查看「专家」（审稿人）相关能力

前置条件：

1. 使用后端认可的 **审稿人** 账号登录用户端（实现上常为 `role === 'reviewer'`，与「发布者 publisher」区分；以前端路由守卫与后端 JWT 为准）。
2. 用户端已启动（本地多见 **http://127.0.0.1:3000**），`VITE_API_URL` 指向可用后端。

典型路径（与当前前端路由一致时）：

1. 登录时在可选角色中选择 **专家 / 审稿人**（若登录页提供该选项）。
2. 侧栏或底部导航进入 **「人工审核」**（路由 **`/review`**），查看任务池（**FR-YHSH-0001**：筛选含流程状态、管理端门闸等）。
3. 点击任务进入 **`/task/detail/:manual_review_id`**：查看协作上下文与 **「查看 AI 检测报告摘要」**，按维度评分与「造假 / 真实」判定后 **提交审核**（**FR-YHSH-0002**）；提交成功后返回任务池。
4. 主页 **`/`** 对审稿人展示「专家 · 人工审核任务池」入口卡片（与发布者主页分区）。

发布者侧「申请复核 / 查看审核进度」与结果展示，见总览中 **人工审核（发布者侧）** 接口列表及 `history.vue`、`manual-review-result.vue` 等页面说明。

## 3. 与学术检测上传页的关系

- **`/upload`（学术检测）**：仅承载 **快速 / 精准** 检测模式与多类型文件上传；论文侧支持的格式与需求 **学术内容检测**（如 `03-5-content-detection.tex` 中对 PDF、DOCX、压缩包等的表述）对齐展示即可。
- **审稿工作**：不通过「再选一个检测模式叫专家」完成，而通过 **人工审核模块** 的任务流转完成。

## 4. 接口契约（后端实现 · 与管理端、发布者协作）

以下路径均相对于 **`/api/`**。

| 优先级 | 接口 | 后端需保证的行为 / 响应要点 |
|--------|------|-----------------------------|
| P0 | `GET /get_reviewer_tasks/` | 校验 **`role === 'reviewer'`**。支持查询参数：`page`、`page_size`、`query`（发布者用户名前缀）、`status`（本人 `ManualReview`：`undo`/`completed`）、`start_time`、`end_time`、**`review_request_status`**（`ReviewRequest.status1`：`pending`/`in_progress`/`completed`）、**`admin_gate_status`**（`ReviewRequest.status2`：`pending`/`accepted`/`refused`）。每条结果建议含：`manual_review_id`、`manual_review_time`、`publisher_username`、`publisher_avatar`、`image_count`、`status`（本人人工审核进度）、**`review_request_id`**、**`review_request_status`**、**`admin_gate_status`**、**`task_kind`**（当前主线为 `image`，便于后续扩展论文/Review）。分页字段与前端约定：`results`、`current_page`、`total_pages`、`total_count` 等。 |
| P0 | `GET /get_review_detail/<manual_review_id>/` | 仅允许该 **`ManualReview.reviewer`** 为当前用户。返回 **`imgs`**：`[{ id, url }, ...]`（与逐图检测、`post_review` 中 `img_id` 一致）；保留或并列返回 `image_urls`/`image_ids` 亦可，但 **`imgs` 为前端首选**。返回 **`review_request`**：`id`、`status`（流程 `status1`）、**`admin_gate_status`**（`status2`）、`request_time`；**`ai_detection_result`**（如 `is_fake`、`confidence_score`、`detection_time`）；**`manual_review_status`**（`undo`/`completed`）。 |
| P0 | `POST /post_review/<manual_review_id>/` | 体为既有 **`result[]`**（七维评分、理由、造假判定、标注点等）；成功后更新 `ManualReview` / `ReviewRequest` 等状态（与现有一致）。 |
| P1 | 管理端审批链路 | 组织管理员对 `ReviewRequest.status2` 审批通过后，审稿端列表方可出现有效任务；管理端详情接口 **`GET /get_review_request_detail/<manual_review_id>/`** 与用户端 **`get_review_detail`** 职责分离，**勿混用**。 |
| — | 论文检测 | 与人工审核无关：`POST /paper/aigc/submit/` 等仍以 **`paper_file_id`、`task_name`** 等既有字段为准，**不设**需求外的 `detection_profile`。 |

## 6. 后端工作清单：用户端登录 / 注册（与「专家」注册相关）

**说明**：下列条目由 **后端** 实现与维护；**前端仓库不修改后端代码**，仅依赖接口行为。

| 项 | 建议后端行为 |
|----|----------------|
| 公开接口与 JWT | `POST /login/`、`POST /register/` 等 **不应** 因请求携带 **过期/无效 `Authorization: Bearer`** 而在 **校验邮箱密码之前** 直接返回 401；避免用户无法登录却误以为密码错误。（实现方式不限：例如公开视图豁免 JWT、或对登录请求忽略 Bearer。） |
| 邮箱登录 | 若采用邮箱 + 密码登录：自定义认证后端应对邮箱 **trim**、**大小写不敏感** 查询（如 `email__iexact`），并校验 **`is_active`**。 |
| `authenticate` 调用 | 登录视图中调用 Django `authenticate` 时传入 **`request`**，保证与中间件/后端约定一致。 |
| `AUTHENTICATION_BACKENDS` | 若同时使用 `ModelBackend` 与邮箱后端，**邮箱登录后端应优先于 `ModelBackend`**，避免邮箱登录被错误处理。 |
| 注册与邀请码 | `POST /register/`：**账号角色以邀请码绑定角色为准**；请求体中的 **`role` 须与邀请码角色一致**，否则返回明确 400。注册成功后邀请码 **`is_used`** 核销（或团队约定的等价语义）。注册写入的用户邮箱建议 **规范化（如小写）**，与登录侧一致。 |
| 登录响应 | 成功登录返回的 **`avatar`** 等字段，在头像文件缺失时不应导致 **500**（可返回空字符串或默认 URL）。 |
