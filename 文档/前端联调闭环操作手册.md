# 前端联调闭环操作手册

本文给出**可照着做**的终端命令（逐条）、浏览器操作（逐步），以及一条完整业务线：**用户端编辑（发布者）→ 管理端审批 → 用户端专家（审稿人）→ 再回到编辑查看结果**。  
更偏「环境 / 端口 / 脚本参数」的说明见《[本地启动与停止手册](./本地启动与停止手册.md)》；接口契约见《[前后端对接-总览](./前后端对接-总览.md)》**第 3 节**。  
**同步说明**：`start-local-windows.ps1` 的开关、用户端 Mock 环境变量、通知 WebSocket 与 Vite `/ws` 代理等，与《本地启动与停止手册》**第 7.1 节**一致；若本手册与仓库内脚本不一致，以 `scripts/start-local-windows.ps1` 为准。

---

## 0. 先选定联调模式

| 模式 | 适用场景 | 你需要起哪些进程 |
|------|----------|------------------|
| **甲：真实后端四服务** | 与 Django、AI、双前端一起联调；WebSocket、轮询、异步任务走后端 | 一键脚本或 4 个终端：AI、Django、用户端、管理端 |
| **乙：全栈前端 Mock** | 本机不跑或不稳定跑 Django；验证界面与人工审核 Mock 闭环 | 至少用户端 dev；管理端可选；**人工审核 Mock 数据在用户端浏览器 localStorage** |

下面**第 1 节**写模式甲的终端命令；**第 2 节**写模式乙；**第 3 节**写「编辑 → 管理端 → 专家 → 编辑」闭环（甲为主，乙在关键步骤注明差异）。

**账号约定**（来自 `scripts/start-local-windows.ps1` 默认种子，若你未改参数）：

| 角色 | 用途 | 邮箱 | 密码 |
|------|------|------|------|
| 组织管理员 | 管理端登录 | `admin@mail.com` | `Admin123!` |
| 发布者（编辑） | 用户端「编辑」 | `publisher_test@example.com` | `Publisher123!` |
| 审稿人（专家） | 用户端「专家」 | `reviewer_test@example.com` | `Reviewer123!` |

**重要**：用户端同一账号**同时只能是一种角色**（`publisher` / `reviewer`）。闭环里要在两种身份间切换时，请**退出登录再登录**，或使用**两个浏览器**（例如 Chrome + Edge，或普通窗口 + 无痕窗口）分别保持「编辑」与「专家」，减少来回登出。

**访问地址**（默认端口）：

- 用户端：`http://127.0.0.1:3000`
- 管理端：`http://127.0.0.1:3001`
- Django：`http://127.0.0.1:8000`
- AI：`http://127.0.0.1:8010`

### 0.1 PowerShell 与一键脚本参数

- **执行策略**：若提示「无法加载，因为在此系统上禁止运行脚本」，在 PowerShell 中执行一次（当前用户即可）：
  ```powershell
  Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
  ```
- **`start-local-windows.ps1` 常用参数**（与脚本 `param` 一致）：

| 参数 | 含义 |
|------|------|
| `-BindHost` | 默认 `127.0.0.1`；各服务监听地址 |
| `-UserPort` / `-AdminPort` / `-BackendPort` / `-AiPort` | 用户端、管理端、Django、AI 端口 |
| `-ForceDependencyRefresh` | 强制刷新依赖（首次或依赖异常时用） |
| `-FullFrontendMock` | 用户端启用全栈前端 Mock（见第 2 节） |
| 种子账号相关 | `-AdminEmail`、`-PublisherEmail`、`-ReviewerEmail` 等可覆盖默认邮箱与密码 |

## 1. 模式甲：真实后端 —— 终端逐条怎么做

以下均在 **PowerShell** 中执行；`<仓库根>` 请换成你的本机路径（例如 `G:\大学\大三下\软工\Academic-plagiarism-Detector`）。

### 1.1 一条命令拉起全套（推荐）

1. 打开 **PowerShell**。
2. 进入仓库根目录：

```powershell
cd <仓库根>
```

3. 执行（首次可不加参数；依赖异常时加 `-ForceDependencyRefresh`）：

```powershell
.\scripts\start-local-windows.ps1
```

4. 等待脚本打印绿色行 **`Local validation environment is ready:`** 及下方各服务 URL 与账号。若加了 `-FullFrontendMock`，在账号块之后另有一行黄色英文 Mock 提示（与 **第 2.1 节**所列一致）。

5. 新开一个 PowerShell，可选自检（与《本地启动与停止手册》一致）：

```powershell
Invoke-RestMethod http://127.0.0.1:8010/health
(Invoke-WebRequest -UseBasicParsing http://127.0.0.1:8000/admin/).StatusCode
(Invoke-WebRequest -UseBasicParsing http://127.0.0.1:3000/).StatusCode
(Invoke-WebRequest -UseBasicParsing http://127.0.0.1:3001/).StatusCode
```

### 1.2 分四个终端手动启动（与脚本等价时）

若不用脚本，可按《本地启动与停止手册》顺序开 **4 个** PowerShell 窗口，依次：

**终端 1 — AI**

```powershell
cd <仓库根>\代码\AI服务\AI服务器代码
# 若尚无最小模型，先训练一次：
# python train_minimal_baseline.py
python ai_http_service.py --host 127.0.0.1 --port 8010
```

**终端 2 — Django**

```powershell
cd <仓库根>\代码\后端\后端代码
.\.venv\Scripts\Activate.ps1
$env:DJANGO_SETTINGS_MODULE="fake_image_detector.local_settings"
$env:AI_SERVICE_URL="http://127.0.0.1:8010"
$env:AI_SERVICE_TIMEOUT_SECONDS="120"
python manage.py runserver 127.0.0.1:8000
```

**终端 3 — 用户端**

```powershell
cd <仓库根>\代码\前端\前端用户端
npm install
npm run dev -- --host 127.0.0.1 --port 3000 --strictPort
```

**终端 4 — 管理端**

```powershell
cd <仓库根>\代码\前端\前端管理端
npm install
npm run dev -- --host 127.0.0.1 --port 3001 --strictPort
```

用户端、管理端 `.env` 中 `VITE_API_URL` 应指向 `http://127.0.0.1:8000`（一键脚本会自动写入）。

## 2. 模式乙：全栈前端 Mock —— 终端与配置

适用于：**不依赖 Django 也能走完界面与人工审核 Mock 闭环**（检测类接口多为前端桩；人工审核数据在**用户端**浏览器 localStorage）。

### 2.1 一条命令（仍起四服务，但用户端写 Mock 环境变量）

```powershell
cd <仓库根>
.\scripts\start-local-windows.ps1 -FullFrontendMock
```

脚本会：

1. 在步骤输出中提示 **`User frontend: enabling VITE_USE_FULL_FRONTEND_MOCK`**（与 `Write-Step` 一致）。就绪后终端会多一行黄色提示：  
   `User client is in full frontend mock mode. Login without Django. Notifications use HTTP polling.`
2. 把**用户端** `.env` 设为：`VITE_USE_FULL_FRONTEND_MOCK=true`，且 `VITE_API_URL` 置空（走 `/api` 相对路径 + 前端 Mock）。

管理端仍指向真实后端地址；若本机未起 Django，管理端除「人工审核 Mock」同源场景外**其它菜单可能报错**，属预期。通知与轮测间隔等行为见《本地启动与停止手册》**第 7.1 节**。

### 2.2 只起用户端（最小）

```powershell
cd <仓库根>\代码\前端\前端用户端
```

在 `.env` 或 `.env.local` 中保证：

```env
VITE_USE_FULL_FRONTEND_MOCK=true
VITE_API_URL=
```

然后：

```powershell
npm install
npm run dev -- --host 127.0.0.1 --port 3000 --strictPort
```

登录页选择「编辑」或「专家」，使用**任意合法邮箱 + 至少 6 位密码**即可登录（Mock 假令牌，见《前后端对接-总览》与 `src/api/user.ts`）。

### 2.3 模式乙下「管理端审批」怎么办

用户端 `:3000` 与管理端 `:3001` 的 **localStorage 不互通**。无后端或未同源 Mock 时，请在用户端打开 **`/annual`**，展开 **「本地联调 · 模拟管理端审批」**，对待审批申请点 **「通过」**，专家任务池才会出现对应任务（详见《前后端对接-总览》**3.5 节**）。

## 3. 闭环主线：编辑 → 管理端 → 专家 → 编辑

目标：完成一次 **「发布者发起人工审核 → 组织管理员通过 → 审稿人提交 → 发布者看到汇总」**。

### 阶段 A：编辑（发布者）— 准备检测任务 ID 并发起点

1. 打开浏览器，访问 `http://127.0.0.1:3000/login`。
2. 在页面上方 **「编辑 / 专家」** 开关中选 **「编辑」**。
3. 使用 **发布者** 账号登录（模式甲：`publisher_test@example.com`；模式乙：任意邮箱 + 密码 ≥6）。
4. 进入 **「统一学术检测」**（侧栏或路由 `/upload`）：上传至少一个资源并完成本批（模式甲走真实接口；模式乙为 Mock 进度）。记下某一行的 **任务 ID**（`taskId`），或到 **「检测历史」**（`/history`）在已完成任务中复制 **任务 ID**。  
   - 发起人工审核时填的 **「关联自动检测任务 ID」** 必须与该 ID 一致（字符串即可）。
5. 进入 **「人工审核申请」**（`/annual`）。
6. 点击 **「发起人工审核申请」**。
7. 在对话框中填写：
   - **关联自动检测任务 ID**：上一步的 `task_id`；
   - **任务类型**：与真实任务一致（下拉项对应 `image_detection` / `paper_aigc` / `resource_check` / `review_detection`）；
   - **申请理由**：不少于约 10 字的说明；
   - **优先级 / 批次 ID**：按需。
8. 点击 **「提交申请」**。列表中应出现一条 **待审批 / 进行中** 状态（具体文案以后端或 Mock 为准）。

**模式乙补充**：提交后若列表为「待管理审批」，请展开 **「模拟管理端审批」** 对该行点 **「通过」**（见上文 **第 2.3 节**）。

---

### 阶段 B：管理端 — 组织管理员审批通过

1. **新开一个浏览器窗口**（或第二个浏览器），访问 `http://127.0.0.1:3001/login`。
2. 使用 **管理员** 登录（模式甲：`admin@mail.com` / `Admin123!`）。
3. 在侧栏或首页卡片进入 **「人工审核申请审批」**（路由 **`/reviews`**）。
4. 在列表中找到刚才发布者提交的那条申请（可按时间、发布者账号辨认）。
5. 打开 **详情**，确认 `detection_task_id`、`task_type`、`reason` 等字段。
6. 选择 **「通过」**（或界面等价操作）。成功后应提示进入专家分配队列；后端会生成 **`manual_review_id`**（Mock 同样会模拟）。

**模式乙**：若未起管理端或未连上后端，**不要跳过**阶段 B 在真实联调中的语义；请回到用户端 `/annual` 用 **「模拟管理端审批」** 完成等价操作。

---

### 阶段 C：专家（审稿人）— 领任务并提交结论

1. 回到 **用户端** `http://127.0.0.1:3000`：若当前仍是发布者账号，请点击 **「退出登录」**。
2. 在登录页将 **「编辑 / 专家」** 选为 **「专家」**。
3. 使用 **审稿人** 账号登录（模式甲：`reviewer_test@example.com`；模式乙：可再注册/登录另一 Mock 专家账号，或同一浏览器先退再登——注意 Mock 下角色由你选的「编辑/专家」与写入的 profile 一致）。
4. 进入 **「人工审核」**（侧栏 **`/review`**）：任务池中应出现 **管理端已通过** 的那条（`manual_review_id` 对应一行）。
5. 点击进入 **`/task/detail/<manual_review_id>`** 详情页，按页面要求完成 **图像 / 论文片段 / Review 文本** 的复核操作（随 `task_type` 不同，界面字段不同）。
6. 提交专家结论（调用 `POST /post_review/<manual_review_id>/`；Mock 下会更新为已完成）。

---

### 阶段 D：再回到编辑 — 查看进度与汇总

1. 在用户端点击 **「退出登录」**。
2. **「编辑 / 专家」** 切回 **「编辑」**，用 **发布者** 账号重新登录。
3. 打开 **`/annual`**：对应申请进度应反映 **专家已提交**（例如进度从 `0/1` 变为 `1/1` 或状态为已完成，以后端/Mock 为准）。
4. 打开 **`/history`**，进入该检测任务的详情：若已实现 `GET /manual-review-requests/by-detection-task/`，可看到关联的 **`review_request_id`** 等。
5. 在 **`/history`** 进入该任务的 **检测记录详情** 页：当专家已提交且接口返回可查看时，按钮 **「查看人工审核结果」** 会可用；点击后会跳转 **`/manual-review-result`** 并带上 `review_request_id` 等查询参数。应能看到 **发布者汇总**（`GET /manual-review-requests/<id>/publisher-summary/`；Mock 为本地桩数据）。  
   - 若按钮仍为禁用，说明 `manual_review_status` 尚未为 `completed` 或 `by-detection-task` 未找到申请，请回到 `/annual` 核对状态。

至此，**编辑 → 管理端 → 专家 → 编辑** 闭环走完。

## 4. 联调时建议顺带验证的能力

1. **通知**  
   - 模式甲：登录用户端后，浏览器控制台应能建立 **`ws://…/ws/notifications/`**（或通过 3000 端口 Vite 代理）；铃铛未读数另有 **HTTP 轮询**（约 60s）。  
   - 模式乙：不建 WebSocket，仍有 **轮询**（约 45s）刷新未读（Mock 数据）。

2. **异步任务轮询**  
   - 在 **`/detect/paper`** 使用非「仅 Mock 瞬时完成」的路径时，可对 **`GET /paper/tasks/{id}/status/`** 多点几次「同步状态」；若开启 Mock 分阶段，数秒内可见 **`in_progress` → `completed`**。  
   - Review 专项 **`/detect/review?task_id=…`** 拉状态同理。

3. **路由守卫**  
   - 专家账号访问 `/upload`、`/annual` 等会被重定向到 **`/review`**；编辑访问 `/review` 会到 **`/upload`**。若与你预期不符，先检查登录角色与顶部身份切换。

## 5. 停止服务

在仓库根目录：

```powershell
.\scripts\stop-local-windows.ps1
```

或在各终端对 AI / Django / 两个 Vite 进程 **Ctrl+C**。详见《本地启动与停止手册》**第 8 节**。

## 6. 常见问题（脚本与终端）

1. **`start-local-windows.ps1` 报 ParserError /「字符串缺少终止符」**  
   多为脚本内字符串引号被误改或混用旧版本。请使用仓库当前版 `scripts/start-local-windows.ps1`（Mock 分支已避免在单引号串内写分号等特殊字符）；本机若曾手改脚本，可 `git checkout -- scripts/start-local-windows.ps1` 恢复。若编辑器保存编码异常，建议该脚本以 **UTF-8** 保存。

2. **路径含中文**  
   PowerShell 对 `cd` 到含中文路径一般可用；若遇编码问题，可先 `chcp 65001` 再执行脚本。
